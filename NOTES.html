1. Fact vs. dimension tables

Concept The test: does this value change per transaction, or stay fixed no matter how many times something is bought/sold?
Gotcha — numeric ≠ fact product_weight_g is a number, but it's still a dimension column — weight doesn't change depending on how many times the product sells. The test is "does it change per transaction," not "is it a number."
My mistake I called review_score a dimension. Wrong — it's a fact, because it's generated because a specific order happened, and it can differ every time the same customer buys something.

2. Star schema (my design)

The grain of fact_order_items — one row = one product line item within one order — is the single most important design decision. Everything else follows from it.

Why not one flat table? Once payments and reviews (different grain — one row per payment or review, not per order item) enter the picture, merging them into the order-item-level table causes double counting on aggregation. Keep different grains as separate fact tables, join via order_id only when needed.

3. Data types that matter

SituationRight typeWhy
Money (price, total_sales)DECIMAL(10,2)Floats store approximate binary values — summing thousands introduces rounding errors. DECIMAL is exact.
IDs / zip codesVARCHARNever do math on them; INT silently strips leading zeros (01037 → 1037).
Dates with a time partDATETIMEDATE truncates the time off entirely.

4. LOAD DATA INFILE gotchas

Error I hit — 1290 The MySQL server is running with the --secure-file-priv option so it cannot execute this statement

Two separate causes, both gave this exact same error:
  1. File wasn't in the trusted folder (find it: SHOW VARIABLES LIKE 'secure_file_priv';)
  2. Path used backslashes \ and double quotes — MySQL needs forward slashes / and single quotes '...', even on Windows
-- Correct format:
LOAD DATA INFILE 'C:/ProgramData/MySQL/MySQL Server 8.0/Uploads/file.csv'
INTO TABLE my_table
FIELDS TERMINATED BY ','
ENCLOSED BY '"'
LINES TERMINATED BY '\n'
IGNORE 1 ROWS;

5. Handling missing values on load

Error I hit — 1366 / 1292 Incorrect integer value: '' for column 'product_name_length'
Incorrect datetime value: '' for column 'order_delivered_carrier_date'

I assumed dates auto-convert blanks to NULL. Wrong, at least under strict mode (the modern MySQL default) — empty string doesn't match the expected format for either numbers or dates, so it errors instead of silently converting. 
LOAD DATA INFILE '...'
INTO TABLE my_table
...
(col1, col2, @temp_col3)
SET col3 = NULLIF(@temp_col3, '');
Text vs numeric columns behave differently Text columns don't error on blank CSV fields — they just insert an empty string '', which is not the same as NULL. If you want missing categories to read "unknown" instead of blank, fix it afterward: UPDATE ... WHERE col = '' OR col IS NULL.

6. Safe update mode

Error I hit — 1175 You are using safe update mode and you tried to update a table without a WHERE that uses a KEY column.

Workbench blocks UPDATE/DELETE unless the WHERE filters on the primary key — even if the query logic is completely valid otherwise.

SET SQL_SAFE_UPDATES = 0;  -- session-only, resets on reconnect

7. UPDATE ... JOIN pattern

Concept Used to backfill one table's column using a lookup in another table — e.g. pulling English category names into dim_products from a translation table. 
UPDATE dim_products p
JOIN category_translation t ON p.product_category_name = t.product_category_name
SET p.product_category_name_english = t.product_category_name_english;

Reads as: "for every row in the left table, find the matching row in the joined table, and copy a value across."

8. Foreign keys

FOREIGN KEY (customer_id) REFERENCES dim_customers(customer_id)
Why this matters Not just documentation — MySQL actively rejects any row whose customer_id doesn't already exist in dim_customers. Free data-quality check on load.

9. Staging → fact table (ELT pattern)

Concept LOAD DATA INFILE reads only one file at a time. The fact table needs data merged from two source CSVs — so raw data is staged first, then combined with SQL JOIN. This is the standard ELT (Extract, Load, Transform) pattern. 
INSERT INTO fact_order_items (...)
SELECT oi.order_id, ..., o.customer_id, o.order_purchase_timestamp, ...
FROM stg_order_items oi
JOIN stg_orders o ON oi.order_id = o.order_id;

10. Business query patterns

Top N

SELECT group_col, SUM(measure) AS total
FROM table
GROUP BY group_col
ORDER BY total DESC
LIMIT 10;

Grouping by two columns

Needed whenever a column's values could repeat across a different context — two cities with the same name in different states, or month "9" appearing in both 2016 and 2017. 
GROUP BY col_a, col_b

Bringing in a dimension column

FROM fact_order_items f
JOIN dim_customers c ON f.customer_id = c.customer_id
GROUP BY c.customer_city
